## 1. Current behavior summary

**Backend** (`core/mitra_views.py:3545–3790`, `core/journey_views.py:1480–1678`): The Day-14 endpoint `POST /api/mitra/journey/checkpoint/14/submit/` accepts exactly `{continue_same, deepen, change_focus}`. `continue_same` and `deepen` copy `cycle_mantra_id / cycle_sankalp_id / cycle_practice_id` verbatim onto the new journey row (lines 3672–3674, 3718–3720) — no call to `generate_triad` or `select_altered_content`. `change_focus` marks the old journey `completed` but creates no new journey inline; it returns `{"nextStep": "orientation"}` (line 3785), pushing the user into the full onboarding flow. No cycle classification enum is emitted.

**Frontend** (`kalpx-app-rn/src/blocks/CheckpointDay14Block.tsx`, `CycleReflectionResultsBlock.tsx`, `engine/actionExecutor.ts:2548–2640`): The modern content-sovereign block renders three CTA slots sourced from the content spine (`M25_checkpoint_day_14`). After submit, FE always routes to `cycle_transitions/checkpoint_results`; the results block then routes `change_focus` to `choice_stack/discipline_select` (a short focus-picker), **bypassing the BE's `nextStep: orientation` hint**. Redux holds a `checkpoint_completed` flag that `Home.tsx:368–401` reads on app resume to auto-route back to an unfinished checkpoint — but this flag is not persisted to AsyncStorage, so an app kill mid-decision can drop the user.

## 4. State safety

**Mid-session kill:** The FE's Redux `checkpoint_completed` flag drives the auto-resume (`Home.tsx:368–401`). If the user kills the app after tapping a decision button but before the response lands, the flag may not persist across app kill — no AsyncStorage backup observed (`kalpx-app-rn/src/engine/cleanupFields.ts:50–56` clears flow-local state on submit). Backend has no counterpart flag to cross-check.

**Long absence (≥48h, <30d):** BE `journey_status` (`mitra_views.py:4493`) returns `hasActiveJourney=true` and the old `journey_id`. FE resume effect re-renders the Day-14 checkpoint if Redux still has `checkpoint_day=14` + `checkpoint_completed=false`. Works in the happy path; breaks if Redux was cleared.

**Long absence (≥30d):** BE `journey_status` (`mitra_views.py:4495–4527`) marks the journey `expired`, emits `welcomeBack=true`. This collapses Day-14 into a generic welcome-back — the unresolved boundary is silently dropped. **This is the hardest fail.** The user who leaves at Day 14 and returns a month later does not get their boundary decision back; they get a "welcome back" screen that treats them like a returning user with no pending decision.

**Welcome-back routing:** No code path today checks "was Day-14 started but not resolved?" before dispatching to the generic welcome-back flow.

## 5. Required changes (minimum to match contract + unblock T2b)

**P0 — blocks T2b:**
1. **Add derivable `day14_started_but_unresolved`** — either
   - (a) new field on Journey: `day14_checkpoint_presented_at = DateTimeField(null=True)`, set when `GET /checkpoint/14/` is called, cleared on successful submit; OR
   - (b) synthesize in the endpoint: `journey.status == "active" AND elapsed_days >= 14 AND not day14_decision_submitted` where the submit flag is a new `decision_submitted_at_day14` boolean.
   Recommend (a) — one migration, one flag set/clear, zero ambiguity.

2. **Welcome-back must respect unresolved boundary.** Add a pre-check in `journey_status` (`mitra_views.py:4469+`) and `welcome_back` (`mitra_views.py:4630+`): if `day14_checkpoint_presented_at is not None and decision_submitted_at_day14 is False`, return `welcomeBack=true + reentry_target: "day14_boundary"` so the FE re-enters the checkpoint instead of the generic welcome screen. This directly enables the T2 `unresolved_day14_boundary` pseudo-tier.

**P1 — quality for Day-14 launch (not T2b blockers):**

3. **Change_focus → short reorientation contract.** Pick one:
   - (a) BE changes line 3785 response to `{"nextStep": "reorientation"}` and adds a new short-reorientation endpoint/flow that surfaces 2–4 questions, not 8.
   - (b) Formalize the current FE override (`choice_stack/discipline_select`) as the ratified contract; document that the BE's "orientation" hint is deprecated; ship a follow-up BE fix.
   Recommend (a) — aligns BE hint with ratified behavior; removes the FE-override fragility.

4. **Cycle classification emit.** Add `cycle_classification: "rebuilding" | "finding_footing" | "holding_steady" | "ready_to_deepen"` to the Day-14 checkpoint GET response, derived from existing signals (engagement_level + completion_rate + support stability). No new signals needed — just a labeling function.

5. **Preserve path_intent across deepen/continue.** In `journey_checkpoint_submit`, when continue_same or deepen fires, explicitly copy `JourneyContext.path_intent` forward to the new cycle so the T1 triad selector doesn't re-derive and drift.

6. **FE AsyncStorage checkpoint guard.** Before `Home.tsx:368–401` resume check, backup `{checkpoint_day, checkpoint_decision_pending}` to AsyncStorage. On app boot, read the key before hydrating Redux — resume takes precedence over dashboard. Prevents mid-session state loss.

7. **FE triad-identity validation.** Post-submit, before navigating to `companion_dashboard/day_active`, re-fetch `/journey/companion/` and compare triad IDs against pre-submit values. Log a warning if mismatch detected on continue_same/deepen.

8. **Deprecate legacy `CycleReflectionBlock`.** It's TSX-coupled and only reachable via schema fallback. Either migrate it to the content spine or document it as removed.

## 6. Recommendation

**`needs_targeted_fixes`** — not a rewrite.

The target contract is **mostly met structurally**. The five invariants around identity preservation (no_auto_advance / no_auto_regenerate / deepen_keeps_same / continue_keeps_same / change_focus_is_explicit) **all pass**. The gaps are concentrated in two places: (1) the **unresolved boundary signal is not derivable**, which is the specific blocker for T2b; (2) the **30-day absence path silently drops Day-14**, which is the highest user-harm bug.

For T2b specifically: P0 items 1 + 2 (new `day14_checkpoint_presented_at` field + welcome-back pre-check) are the full set of plumbing needed. ~1 backend PR, one migration, one small patch to welcome-back. Neither FE change is required for T2b to work — the FE already resumes correctly when Redux is intact.

**Proposed next step:** ship P0 items 1 + 2 as a small follow-up patch, then proceed with T2b wiring. P1 items can land as a separate PR without blocking the router endpoint.
